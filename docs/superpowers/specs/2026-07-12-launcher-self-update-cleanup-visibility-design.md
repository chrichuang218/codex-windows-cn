# 启动器自更新清理与状态可见性设计

## 背景

启动器自更新会先把正在运行的 `codex-launcher.exe` 重命名为
`codex-launcher.old.exe`，再把已校验的新版本移动到标准路径。Windows 不允许直接
删除正在运行的可执行文件，因此交换阶段必须保留旧文件。

当前实现把旧文件当作永久回滚文件，后续启动也不清理，导致安装目录长期残留
`codex-launcher.old.exe`。同时，启动器已经在应用初始化时后台检查 GitHub Release，
但设置页入口一直显示固定文案，用户只有进入二级页面才能知道检查结果。

## 目标

- 保留自更新交换阶段的失败回滚能力。
- 在新启动器正常启动后自动清理旧启动器文件。
- 无需进入二级页面即可看到启动器更新状态。
- 有可用更新时提供足够明显但不过度打扰的提示。

## 非目标

- 不增加手动回滚功能或长期版本归档。
- 不改变 ChatGPT 桌面应用的更新流程。
- 不增加新的更新源、代理、镜像或诊断能力。
- 不把启动器更新与 ChatGPT 更新合并为同一个状态。

## 方案

### 旧启动器清理

保留现有下载、校验、冒烟测试和原子交换顺序。`--self-test` 必须在任何清理前直接
退出，避免候选程序在正式替换前删除可回滚文件。

正常图形界面启动时，在单实例检查和 Tauri 初始化前执行一次启动器残留清理：

- 删除 `codex-launcher.new.exe`，清理由中断更新留下的候选文件。
- 删除 `codex-launcher.old.exe`，清理由上一次成功更新留下的旧程序。
- 删除失败不阻止启动。文件可能仍被旧进程或安全软件占用，后续每次正常启动继续重试。
- 命令行辅助模式不执行这项清理，避免一次性提权或卸载辅助进程产生额外副作用。

清理实现接受明确的目录参数，使文件行为可在临时目录中确定性测试；生产入口只负责
传入当前可执行文件所在目录。

### 更新状态可见性

后台检查时机保持不变。设置页的“启动器更新”卡片直接消费现有
`launcherUpdateStatus`，按状态显示：

- 检查中：`正在后台检查更新`
- 有更新：`发现 vX.Y.Z，点击更新`
- 已是最新：`已是最新版本 vX.Y.Z`
- 检查失败：`检查失败，点击查看`
- 跳过或关闭检查：使用后端返回的简短状态信息

有可用更新时：

- 卡片使用现有界面色系增加高亮状态，不改变整体布局。
- 左侧“设置”入口显示一个小提示点，使用户在其他页面也能发现更新。
- 点击卡片仍进入现有启动器更新页面，继续复用发布页、延后和应用更新操作。

顶部 ChatGPT 更新提示保持原样，避免用户误以为启动器版本就是 ChatGPT 版本。

## 错误处理

- 启动清理属于非关键维护动作，失败时保留文件并在下次启动重试，不伪报清理成功。
- GitHub 检查失败时，设置卡片明确显示失败状态；详细错误仍在二级页面展示。
- 更新下载、校验、冒烟测试或交换失败时，沿用现有显式错误事件和回滚逻辑。

## 测试

### Rust

- 临时目录同时包含 `.new.exe` 和 `.old.exe` 时，清理函数删除两者。
- 文件不存在时清理函数可重复调用。
- 受占用或不可删除文件不影响启动路径，并可在后续调用重试。
- `--self-test` 仍在清理前退出，由入口顺序测试或现有自测试契约覆盖。

### 前端

- 后台检查返回可用更新后，设置卡片直接显示目标版本。
- 有可用更新时设置导航显示提示点。
- 已是最新、检查中和失败状态显示对应文案。
- 点击卡片仍能进入现有启动器更新页面并应用更新。

## 验收标准

- 完成一次启动器更新并关闭旧进程后，下一次正常启动会删除
  `codex-launcher.old.exe`。
- 清理失败不会阻止启动，后续启动仍会重试。
- 用户打开设置页即可判断启动器是否有更新，无需先进入二级页面。
- 用户位于概览或版本页时，有可用启动器更新会在“设置”入口显示提示点。
- Rust 与前端针对性测试、完整测试和生产构建通过。
